Helix 24.03 版本来了！首先，**非常感谢**所有让这个版本成为可能的人。这个版本包含了来自 125 位贡献者的更改。

刚接触 Helix？
Helix 是一个模式文本编辑器，内置支持多重选择、语言服务器协议（LSP）、tree-sitter，以及实验性的调试适配器协议（DAP）支持。

让我们来看看这个版本的亮点。

## Amp 风格的跳转

{{ asciinema(id="amp-jump", width="94", height="25") }}

跳转功能在 (Neo)Vim 插件领域很受欢迎，甚至有插件将相同的功能添加到浏览器等其他工具中。它们允许您在视图的大部分区域高效地移动选择，就像用鼠标点击一样，但通过输入"标签"来实现。24.03 引入了受 [Amp 编辑器]的[跳转模式]启发的跳转命令。按下 `gw` 添加跳转标签，然后输入其中一个标签即可跳转到该标签下的单词。在选择模式（`v`）中使用 `gw` 会扩展选择。

## 块注释

过去，Helix 只能切换行注释，如 `//` 和 `#`，而像 OCaml 这样的语言只能使用 `(*` 这样的"行"注释标记作为变通方案。在 24.03 中，Helix 现在也可以切换块注释了。使用 `C-c` 或 `<space>c` 根据语言的注释令牌配置智能地添加或移除当前选择周围的行注释或块注释，使用 `<space>c` 切换当前选择周围的块注释，或使用 `<space><A-c>` 仅切换当前行的行注释。

## tree-sitter 注入的改进

Helix 使用 [`tree-sitter`] 增量解析来实现语法高亮、文本对象、缩进以及一些动作和命令。24.03 改进了我们处理_[注入]_的方式——这是一个强大的 tree-sitter 功能，用于解析包含多种语言的文档。例如，在 HTML 的 `<script>` 或 `<style>` 标签中可能包含 JavaScript、CSS 或其他语言。HTML 解析器不需要知道如何解析所有这些语言。相反，它可以_注入_ JavaScript 或 CSS 解析器来处理 `<script>` 或 `<style>` 标签的内容。

{{ asciinema(id="ts-subtree-and-motions-injection", width="94", height="25") }}

其中一项改进是针对 `:tree-sitter-subtree` 命令的，该命令显示光标下语法树的 S-表达式。24.03 显示的是选择下的注入层，而不仅仅是根层。例如，在这个 asciicast 中，我们现在显示文档中 JavaScript 部分的语法树，而过去我们只显示 HTML 部分的树。

另一项重大改进是针对 tree-sitter 动作的。`A-o`（alt + o）将选择扩展到当前选择所覆盖的语法树节点的父节点。`A-i` 收缩到子节点，`A-n` 和 `A-p` 分别转到<em>下</em>一个和<em>上</em>一个节点。以前，这些命令只在根层（例如 asciicast 中的 HTML）上工作，但现在它们会找到包含选择的注入层，并沿着该层的语法树移动。在内部，我们将注入层组织成树状结构，以便这些动作在需要时可以在层之间切换。

请关注未来对注入的改进，例如在注入内容中查找文本对象。

## 内部改进

24.03 还进行了重大的内部改进。首先是来自新的"事件系统"——一个围绕 Tokio 通道和任务构建的系统，允许 Helix 代码库的不同部分相互通信。事件系统还添加了通用的方法来防抖动和在后台线程中运行任务，这样我们就可以防止 UI 卡顿。代码库的某些部分已经迁移到事件系统，如 LSP 补全和签名帮助。在 24.03 中，您会注意到 LSP 补全不会在插入模式下仅仅导航（例如使用方向键）后自动弹出。它现在改为挂钩到文档更改中，因此它可以在您开始更改文档时智能弹出，而不是每次编辑器空闲时都弹出。

另一个值得期待的重大改进是使用 [`regex-cursor`] 全面替换 `regex` crate。`regex-cursor` 是一个流式正则表达式实现，与我们用于表示所有文档的 [`ropey`] crate 兼容。`regex-cursor` 能够对_非连续字符串_——可能不在内存中连续存放的输入——进行操作。在 rope 中，文档的不同部分可能在内部单独存储。为了使用 `regex` API，我们需要将文档的切片转换为常规的 Rust `String`，这会复制该文档切片的内存占用。`regex-cursor` 则直接对 rope 的块进行操作，大大提高了效率。
